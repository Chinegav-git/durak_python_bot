# Статичний аналіз синхронізації стану БД

Цей документ описує результат статичного аналізу кодової бази станом на 2024-07-24.

## Загальний висновок

Програмний код на рівні Python є внутрішньо узгодженим. Залежності між модулями та конфігураціями виглядають коректними.

Однак існує **фундаментальний архітектурний розрив між програмною логікою (Code Layer) та фізичним станом зовнішньої системи (PostgreSQL)**.

Код *очікує*, що таблиці `user`, `chat`, `game`, `chatsetting` існують, але вони ще не були створені у базі даних. Будь-яка функція в коді, яка намагається виконати операцію читання або запису в базу даних, неминуче призведе до збою `UndefinedTableError` на рівні драйвера БД.

Це не помилка в логіці Python, а відсутність необхідного кроку в налаштуванні інфраструктури, який синхронізує очікування коду з реальністю бази даних.

## Детальний аналіз

### 1. Аналіз шару даних (Data Layer)

Цей шар складається з трьох ключових компонентів, які повинні працювати узгоджено:

*   **Компонент А: Моделі (`durak/db/models.py`)**
    *   **Призначення:** Описує *очікувану* структуру даних у вигляді Python-класів (`User`, `Chat`, `Game`, `ChatSetting`). Це є "кресленням" вашої бази даних.

*   **Компонент Б: Конфігурація ORM (`durak/db/tortoise_config.py`)**
    *   **Призначення:** "Міст" між кодом і базою даних. Змінна `TORTOISE_ORM` вказує, де знаходяться "креслення" моделей.

*   **Компонент В: Конфігурація міграцій (`pyproject.toml`)**
    *   **Призначення:** "Міст" для інструменту `aerich`. Вказує `aerich`, де знаходиться конфігурація ORM.

**Висновок по шару даних:** На рівні коду ці три компоненти **узгоджені** та правильно посилаються один на одного.

### 2. Аналіз потоку даних при взаємодії з БД

#### Сценарій: Зміна налаштувань чату (команда `/settings`)

1.  **Компонент:** `durak/handlers/settings.py` отримує запит.
2.  **Взаємодія:** Код виконує виклик `await ChatSetting.get_or_create(...)`.
3.  **Трансляція в SQL:** `Tortoise ORM` генерує SQL-запит `SELECT ... FROM "chatsetting" WHERE id=...`.
4.  **ТОЧКА ЗБОЮ:** SQL-запит відправляється до PostgreSQL. Сервер повертає помилку `UndefinedTableError`, оскільки таблиці `"chatsetting"` не існує.

#### Сценарій: Початок нової гри (команда `/new`)

1.  **Компонент:** `durak/logic/game_manager.py` намагається оновити статус гри в БД.
2.  **Взаємодія:** Код викликає оновлення моделі `ChatSetting`.
3.  **Трансляція в SQL:** `Tortoise ORM` генерує `UPDATE "chatsetting" SET ...`.
4.  **ТОЧКА ЗБОЮ:** З тієї ж причини PostgreSQL відповість помилкою, оскільки таблиці `"chatsetting"` не існує.

### Рекомендація

Для усунення цього розриву необхідно виконати процес міграції бази даних за допомогою `aerich`, щоб створити фізичні таблиці, які відповідають визначеним моделям.
